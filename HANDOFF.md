# Matrix Pi — Project Handoff

## Overview
A 64×64 RGB LED matrix display system controlled by a Raspberry Pi. Users control the display through a web frontend. The Pi connects to a cloud backend via WebSocket and launches display programs as subprocesses.

---

## Architecture

```
[Web Frontend (Next.js)]
        ↕ REST + WebSocket
[Backend (FastAPI on Render)]
        ↕ WebSocket (persistent)
[Pi Agent (agent.py)]
        ↓ subprocess (sudo)
[Display Programs: clock, weather, MLB, music, picture, drawing, text, map]
        ↓
[64×64 LED Matrix via rpi-rgb-led-matrix]
```

## Agent (`matrix-agent/agent.py`)

### Key behaviours
- Connects to backend WebSocket on boot, reconnects automatically on drop
- Receives `state` messages → calls `runner.apply_mode(m)`
- Each mode launches a **subprocess** with `sudo -n` — never directly initialises the LED hardware in-process (doing so causes GPIO/DMA kernel panics)
- `_stop()` sends SIGTERM → waits up to 4s → escalates to SIGKILL. This prevents two processes fighting over LED hardware simultaneously
- Watchdog loop restarts crashed display processes every 5s
- Schedule loop checks time-based mode slots every 30s

### WebSocket message types handled
- `state` — apply mode/brightness/rotation
- `settings` — location, units
- `map_config` — addresses for map mode
- `mlb_config` — fetch+write MLB config, restart if running
- `schedule` — update schedule slots
- `cmd: update` — git pull + reboot

### .env variables
```
BACKEND_BASE=https://...
WS_URL=wss://...
DEVICE_TOKEN=<uuid>
ROTATION=90
HOME_DIR=/home/pi_three
MLB_DIR, MUSIC_DIR, CLOCK_DIR, WEATHER_DIR, PICTURE_DIR, DRAWING_DIR, TEXT_DIR
```

## start.sh

```bash
PYTHON="$SCRIPT_DIR/.venv/bin/python"
# Falls back to python3 if venv not built
$PYTHON wifi_setup.py || true
exec $PYTHON agent.py
```

**Critical:** Must use `.venv/bin/python` not bare `python3`. System Python doesn't have `websockets` installed.

## WiFi Provisioning (`wifi_setup.py`)

### How it works
1. On boot, checks `is_online()` (connects to 8.8.8.8:53)
2. If offline → scans networks, caches results, creates hotspot `Matrix-Setup`/`matrix1234`
3. Writes captive portal DNS config to `/etc/NetworkManager/dnsmasq-shared.d/matrix-captive.conf`
4. Adds iptables redirect: port 80 → 8080
5. Starts HTTP server on port 8080
6. User connects phone/laptop to `Matrix-Setup`, opens `http://10.42.0.1:8080`
7. User selects network, enters password, clicks Connect
8. Pi connects, script exits 0, `start.sh` runs `agent.py`

### Key design decisions
- **Scan BEFORE creating hotspot** — wlan0 can't scan while in AP mode (single WiFi chip)
- **HTTP server on port 8080** — port 80 requires root and had binding issues; iptables redirects port 80 → 8080
- **No RGBMatrix in-process** — directly initialising RGBMatrix in wifi_setup.py causes kernel panics. Must use subprocess
- **Respond to /connect before acting** — hotspot drops when connecting, killing the browser's connection. Backend responds immediately, does work in background thread
- **Wrong password handling** — on failure, deletes bad NM profile so it doesn't persist across reboots, restarts hotspot, stores error in `_server_state["last_error"]`, shown on next page load
- **"Failed to fetch" = success** — if the browser loses connection during /connect, it means the hotspot dropped because we connected successfully

## Setup Script (`setup.sh`)

Run once on a new Pi. Does:
- Installs system packages
- Clones repo to `~/matrix-agent`
- Builds `rpi-rgb-led-matrix` wheel (pinned to `e947417f` — last commit before Pi5 RP1 backend that breaks 32-bit ARM)
- Creates venvs for agent, MLB, Spotify
- Writes `.env` with device token from backend API
- Installs systemd service `matrix-agent`

**Note:** Username on Pi matters. If not `pi_two`, a symlink `/home/pi_two → /home/<user>` is created. Current test device uses `pi_three`.

## Common Issues & Fixes

| Problem | Cause | Fix |
|---------|-------|-----|
| `ModuleNotFoundError: websockets` | start.sh used bare `python3` (system Python) | Use `.venv/bin/python` |
| Pi kernel panic during wifi_setup | RGBMatrix initialised in-process | Never init RGBMatrix in wifi_setup.py; use subprocess only |
| Two display processes fighting over LED | `_stop()` didn't wait for process to die | `p.wait(timeout=4)` + SIGKILL escalation |
| MLB never rotates (no preferred team) | `is_offday_for_preferred_team()` returned True for empty list | Fixed in `schedule.py` |
| Rotation "off" doesn't work | Race condition — old process still running | Fixed by `_stop()` blocking |
| SSH drops when testing wifi_setup | SSH is over WiFi; hotspot takes over wlan0 | Normal — test over ethernet or accept the drop |
| Captive portal infinite reload | Redirect for `/favicon.ico` → reload loop | Serve HTML for all unknown paths (no redirects) |
